# Sharing and Permissions

This document describes how plans can be shared and how access is controlled. It builds on the collaboration model by defining who may participate in editing and viewing a plan.

## Overview

Plans support multiple actors with different access levels. Sharing is a core capability and must integrate safely with realtime collaboration.

## Sharing mechanisms

### Email invitations

- Admins can invite people by email to create or update a membership.
- Invitations grant persistent access as `admin` or `member`.
- Access can be revoked.

### Share links

- Generated by admins.
- Require sign-in; accepting the link adds the user as a `member`.
- Can be revoked or rotated.

### Public access

- Public plans (`plans.is_public`) are readable via `public_slug`.
- Edit access is gated separately via membership or `edit_token`.

## Access levels

- **Owner** – Stored on `plans.user_id`; full control and treated as admin.
- **Admin** – Membership tier that can manage sharing and edit plan content.
- **Member** – Membership tier that can edit plan content but cannot manage sharing.
- **Public visitor** – Read-only access when a plan is public.

## Enforcement model

Permissions are enforced at three levels:

1. **Client** – The UI hides or disables actions based on access level (`canEdit`, `canManageMembers`).
2. **Server/RPC** – Server actions and database functions validate membership/admin status.
3. **Database + realtime** – RLS policies restrict reads/writes on planner tables; realtime honors the same policies.

## Security principles

- Never trust client‑only checks.
- All writes must be validated server‑side.
- Realtime streams must respect permissions.
